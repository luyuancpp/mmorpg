# 玩家线上数据下线导出操作记录

> 日期：2026-04-16
> 主题：将单个玩家的线上实时数据安全地导出到内网隔离环境，用于排查、对比、回档准备或问题复现。

## 核心原则

- 不要直接对正在变化的线上状态做人工处理。
- 如果玩家在线，必须先踢下线或强制落盘，再导出数据。
- 导出应以单玩家为粒度，明确玩家、区服、时间点、操作人和原因。
- 导出的生产数据只能进入隔离影子环境，不应直接污染共享开发环境。
- “拿数据下来”和“把数据改回去”是两件事，必须分开审批和审计。

## 推荐流程

1. 确认 player_id、account_id、zone_id。
2. 判断玩家是否在线；若在线，先做安全落盘或踢下线。
3. 从权威数据源导出：优先快照、数据库或稳定持久化链路。
4. 将数据写入隔离文件或影子库，并记录时间、原因、操作人、校验信息。
5. 在内网环境进行 diff、复现、只读分析或回档预演。
6. 如需恢复，必须走显式导入流程，并保留审计日志。

## 风险点

- 在线态直接抓取可能拿到半更新数据，导致分析误判。
- 未隔离的生产数据容易污染测试环境。
- 单玩家回档不是时间倒流；涉及交易、邮件、公会、拍卖等跨玩家副作用时，需要结合流水判断。

## 结论

这类操作应被视为标准运维动作：

- 先冻结写入
- 再导出权威数据
- 仅在隔离环境分析
- 恢复动作单独审批

这样既能满足排查问题的需要，也能避免对线上造成二次风险。
