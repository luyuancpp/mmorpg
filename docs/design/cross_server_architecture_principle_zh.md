# MMO 跨服架构（全局设计文档）

## 游戏模型
- MMO（魔兽世界类），约 1000 个大区（realm）
- 玩家归属一个主区，但可以自由访问任意大区
- 每个大区拥有各自的 Redis
- 切换场景的延迟可接受；**稳定性与正确性**是最高优先级

## **核心原则：位置透明**
> 不管玩家在哪个服务器上，处理逻辑都和在本服一样，不需要判断"是否跨服"。
> 此原则必须适用于所有未来的跨服相关开发。

- 逻辑层**绝不**根据"该玩家是跨服还是本服"来分支处理
- 所有游戏逻辑对每个玩家一视同仁，不关心其来源大区
- 不向目标服务器同步/复制数据——玩家数据始终留在主区
- 消除：登录时的跨服检测、数据一致性问题、散落各处的 if-跨服分支

## 痛点（历史经验——避免重蹈覆辙）
- 数据同步方案：将数据复制到目标服务器 → 一致性噩梦
- 每个处理点都检查"是否跨服？" → 易出错、难维护
- 登录流程检测跨服状态 → 过度复杂、脆弱

## 选定架构：方案 A——独立数据服务（数据代理微服务）
- 一个专用的 Go（go-zero）微服务，负责所有跨区数据路由
- 场景服务器通过 gRPC 统一接口调用数据服务（如 `GetPlayerBag(player_id)`）
- 数据服务内部解析 `player_id → home_realm_id → 正确的 Redis` 并透明路由
- 场景服务器只有一个 gRPC 地址，完全不知道 Redis 拓扑或大区路由
- 选择理由：**稳定性 > 延迟**——切换场景可容忍多几毫秒；场景服务器内活动部件更少，所有路由复杂度集中在一个服务中，更易维护/调试/更新路由规则

### 为什么不选方案 B（内嵌路由）
- 每个场景服务器连接 1000 个 Redis 实例 = 连接爆炸
- 路由逻辑分散在所有场景服务器实例中 = 更新困难
- 方案 A 集中复杂度；方案 B 分散复杂度

### 数据流
```
玩家 → Gate → 场景服务器（任意大区）
                    │
                    │ gRPC（统一接口，大区无关）
                    ▼
              数据服务（Go, go-zero）
                    │
                    │ 通过 player_id → home_realm_id 路由
                    ▼
              Redis（玩家的主区）
```

### 场景切换流程
```
玩家在大区 1，想要访问大区 500
→ 客户端请求切换大区
→ Player Locator 将玩家的当前 scene_node 更新为大区 500 的场景
→ Gate 将连接路由到大区 500 的场景服务器
→ 大区 500 的场景服务器通过数据服务正常处理
→ 数据服务透明地读写大区 1 的 Redis
→ 零数据迁移，零跨服分支
```

### 数据服务设计要点
- 水平可扩展：按大区范围分片（实例 1 → 大区 1~200，实例 2 → 大区 201~400，以此类推）
- 路由表：`player_id → home_realm_id → redis_addr`（一致性哈希或查找表）
- 在此层统一实现重试/熔断/降级——场景服务器无需担心 Redis 故障
- 可为热点数据（HP、位置、Buff）添加 LRU 缓存以减少跨区 Redis 调用
- 扩展现有 `go/db/` 服务或创建新的 `go/data_service/`

## 区/大区层级
- **区（Zone）**：游戏逻辑层面的服务器，如区 1~10000
- **大区（Region）**：区的分组，如大区 A = 区 1~100
- Redis 按**大区**分配，而非按区：10000 个区 / 每大区 100 个 = 100 个 Redis 集群
- **锁区（Lock Region）**：若大区锁定，玩家只能访问主大区内的区服；若解锁，可自由访问区 1~10000
- 锁区逻辑位于 **Scene Manager** 中，而非数据服务

## 玩家 ID 设计（Snowflake）
- 使用现有 `snow_flake.h`：`[time:32b][node_id:17b][step:15b]`
- **禁止在 player_id 中编码 zone_id**——否则合服时会出问题
- node_id = 生成该 ID 的物理节点，不是 zone_id
- player_id 永久不变，即使经历合服

## 路由：独立映射表
- `player_id → home_zone_id`：全局映射表（独立于游戏 Redis）
- `home_zone_id → region_id`：配置表
- `region_id → redis_cluster_addr`：配置表
- 映射表存储在专用的全局 Redis 中（1 亿玩家约需 2~3 GB）或 MySQL + 本地缓存
- 玩家注册时写入映射：`player_mapping[player_id] = home_zone_id`

## 合服策略
- **player_id 永不改变**——无需迁移 ID 数据
- 方案 1（推荐）：将区 2 的 Redis 数据迁移到区 1 的 Redis，更新映射表（所有区 2 玩家 → 区 1）
- 方案 2（更快）：将 zone_1 和 zone_2 的路由都指向同一个 Redis，导入数据，无需更改映射
- 两种方案均不需要修改任何业务代码

## 数据一致性与顺序保证
- **Redis 单线程保证**：同一 key 在同一连接上 = 严格 FIFO，不会乱序
- **真正的风险在上游并发**：两个场景服务器同时写同一个玩家
- **解决方案：每个玩家单写入者**——一个玩家在任意时刻只在一个场景服务器上
- **场景切换串行化**（由 Scene Manager 保证）：
  1. 通知旧场景："释放玩家 1001"
  2. 等待旧场景确认："已保存并释放"  ← 必须等待
  3. 确认后才通知新场景："加载玩家 1001"
  4. 第 2 步超时 → 中止，不进入第 3 步
- **防御层次**：
  - 第 1 层：Scene Manager 保证串行交接（正常情况）
  - 第 2 层：数据服务的按玩家分布式锁（Redis SETNX，TTL ~3s）处理异常情况
  - 第 3 层：关键数据的乐观锁/版本字段（WATCH+MULTI 或版本列）作为最终兜底

## 关键设计规则
1. **数据服务是唯一知道跨区路由的地方**——其上层全部大区无关
2. **玩家数据永不移动**——始终存在于主区的 Redis/DB 中，单一数据源
3. **不在任何玩家或会话对象上设置"跨服模式"标志**
4. **新功能必须遵循此原则**——如果某功能需要问"这个玩家来自哪个大区？"，说明设计有问题（只有数据服务可以问这个问题）
5. **稳定性 > 延迟**——场景切换可容忍多几毫秒；数据正确性不可妥协
6. **单一路由真相源**——仅在数据服务中修改路由规则，Scene/Gate/Centre 零改动
7. **player_id 与大区无关**——禁止在 ID 中编码区信息，使用映射表
8. **每个玩家单写入者**——不允许不同场景服务器并发写同一玩家

## 与现有架构的集成
- `player_locator` 已在追踪玩家所在的场景——扩展其以追踪 home_zone_id
- Gate + Kafka 命令路由无需改动——Gate 不关心玩家的主区
- Scene Manager 将玩家路由到任意大区的场景服务器——同时保证场景切换串行化
- 扩展现有 `go/db/` 服务作为数据服务，或创建新的 `go/data_service/` 微服务
- 现有 snowflake（`snow_flake.h`）无需修改——禁止嵌入区信息
