# SLG 行军位置公式

## 前置条件
- A* 寻路在行军注册时一次性算出完整路径并存储
- `path = [tile_3, tile_7, tile_11, tile_15, ...]` — 起点到终点的格子序列
- 每个元素是 tile_id (`tile_id = y * MAP_WIDTH + x`)

## 位置公式 (O(1) 查表)
```
elapsed = now - start_time
tiles_traveled = floor(elapsed * speed)
current_tile = path[min(tiles_traveled, path.length - 1)]
```
- 不是寻路，是查表：path[] 预计算好，tiles_traveled 是下标偏移
- 任意时刻 O(1) 算出当前位置，无需逐帧推进

## speed 单位是 格/秒 (tiles/second)

维度分析：`秒 × 格/秒 = 格数（数组下标）`

- 不需要乘格子长度，因为 path[] 里已经是真实 tile_id（世界坐标），下标直接取出位置
- 格子物理长度只在两处需要：
  1. **客户端渲染**：两格之间插值动画需要像素坐标
  2. **A\* 寻路**：计算路径代价时对角线 cost = √2 vs 直线 cost = 1

## 对角线问题：简化 vs 加权

### 简化模型 O(1) — 每步都算 1 格
- speed = 格/秒，对角线和直线花**相同时间**
- 对角线实际距离 √2 ≈ 1.414，视觉上偏快 ~41%
- 适用：大地图缩放远、格子小，玩家感知不到

### 加权模型 O(log n) — 对角线算 √2 格
- 注册时预计算 `cum_dist[]` 累积距离数组：
  ```
  weight[i] = is_diagonal(path[i-1], path[i]) ? 1.414 : 1.0
  cum_dist[0] = 0
  cum_dist[i] = cum_dist[i-1] + weight[i]
  ```
- 查询：`distance = elapsed * speed`，二分查找 `cum_dist[]` 得到下标
- 1000 格路径只需 ~10 次比较，性能无压力
- 视觉速度完全一致

### 对比
| | 简化模型 | 加权模型 |
|--|---------|---------|
| 查询 | O(1) 数组下标 | O(log n) 二分 |
| 额外存储 | 无 | cum_dist[] 数组 |
| 对角线速度 | 视觉偏快 41% | 视觉一致 |
| 适用场景 | 大地图缩放远、格子小 | 近距离可见行军 |

### 决策建议
- **默认先用简化模型**，让策划看实际行军效果
- 如果策划觉得对角线速度不一致需要修正，再切换到加权模型
- 切换成本低：只需加 `cum_dist[]` 预计算 + 查询改二分

## 客户端同步
- 服务器只发送 `{path, start_time, speed}`
- 客户端自行插值显示行军动画
- 仅在速度变化/召回时重新下发
