## 一、ECS 与世界状态

### Q1: MOBA DS 用 ECS 还是面向对象？为什么？

**A:** 现代 MOBA DS **几乎清一色 ECS**（Entity-Component-System），原因：

1. **数据驱动**：英雄/技能/Buff 全部由配表生成 Component 组合，新增英雄不写新类
2. **缓存友好**：System 按 Component 类型批处理，遍历时 SoA 命中 L1/L2 cache
3. **确定性容易保证**：纯数据 + 纯函数 System，没有继承多态隐藏状态
4. **回放/快照轻松**：序列化 Component 数组即全状态
5. **热更**：Component 是 POD，System 是逻辑，可以分别替换

**反面教训（OOP）**：早期端游 MOBA 用 `class Hero : public Unit : public Entity`，最后技能要给小兵用、Buff 要给塔挂、视野要给眼挂...继承树爆炸。

**我们项目** cpp/libs/services/scene 就是 ECS（entt），完全契合（虽然是 MMO 不是 MOBA，但 ECS 思想通用）。参考 [ecs.md](ecs.md) 和 [ecs-component-access-rules.md](ecs-component-access-rules.md)。

### Q2: 一局 MOBA 里 entity 大概多少？怎么避免 N² 遍历？

**A:**

**典型 entity 数量（5v5 MOBA 中盘）：**

| 类型 | 数量 |
|------|------|
| 英雄 | 10 |
| 小兵（3 路 × 双方） | 30–60 |
| 防御塔/基地 | 11 + 11 |
| 野怪 | 30–80 |
| 守卫（眼） | 10–30 |
| 召唤物/分身/小动物 | 20–50 |
| 子弹/投射物 | 20–100 |
| Buff/Debuff 挂载 | 200–500 |
| **合计** | **300–900** |

**避免 N² 的核心：空间索引 + 兴趣过滤。**

| 查询 | 数据结构 |
|------|---------|
| 半径范围内单位（技能命中、AOE） | 均匀网格（uniform grid）/ 四叉树。MOBA 地图 < 16km²，**单层网格（cell = 200unit）** 足够，比四叉树常数小 |
| 直线穿透（弹道）| 沿格子 DDA / Bresenham 遍历，每格只查格内单位 |
| 视野/AOI | 网格 bitmap，前面讲过 |
| 最近敌方（小兵/塔 AI 自动攻击）| 维护 `近敌候选列表`，距离变化超阈值才重排 |
| 仇恨/嘲讽 | 每个单位维护小的 `attackers` 列表，O(几) |

**关键**：MOBA 单位数虽然不算大，但 **AOE 技能 / 弹道 / 视野 / AI** 每帧都要做空间查询，没有索引每帧轻松上百万次比较 → 帧时间炸穿。

### Q3: Buff / Debuff 系统怎么设计？时间到期、叠层、互斥怎么处理？

**A:**

**核心结构：**

```cpp
struct BuffInstance {
  uint32 buff_id;          // 配表 ID
  uint32 source_entity;    // 释放者
  uint32 target_entity;    // 受体
  int32  stacks;           // 当前叠层
  int64  end_frame;        // 到期帧（确定性时间）
  int64  next_tick_frame;  // 下次结算（DOT/HOT）
  // ...
};

struct BuffContainer {  // 挂在目标 entity 上
  std::vector<BuffInstance> buffs;
};
```

**关键操作的策略表：**

| 操作 | 策略（由配表决定） |
|------|------------------|
| 同 buff_id 同来源再次施加 | refresh（刷新时间）/ stack（叠层）/ ignore / replace |
| 同 buff_id 不同来源 | independent / overwrite_if_stronger |
| 互斥组（如硬控不叠加） | 维护 `category` 字段，同 category 只保留最强 |
| 到期 | 按 `end_frame` 优先队列，每帧 pop |
| 周期结算（DOT） | 按 `next_tick_frame` 优先队列 |
| 被驱散 | 按 dispel_type 过滤（控制类/魔法类/物理类） |

**优化：**
- 每个 BuffContainer 内部 buff 数通常 < 10，线性查找比 hash 快
- 全局两个优先队列：到期、tick，避免遍历所有单位的所有 buff
- **属性修正**用 dirty 标记 + 重算：buff add/remove 时把 entity 的 attribute_dirty=true，下一次属性读取时重算所有 buff 影响的最终值（不要每次属性变化都广播）

**坑：buff 在 OnAdd / OnRemove 里又施加另一个 buff** → 必须用 deferred event 队列，不能在遍历中改容器。

### Q4: 技能系统怎么设计？怎么数据驱动？

**A:** **技能 = 阶段机 + 效果列表。**

**技能配表结构：**

```yaml
skill_id: 10001
name: 火球术
cast_type: TARGET_POINT    # 指向点 / 指向单位 / 自身 / 无目标
range: 800
cast_time: 0.3s             # 前摇
cooldown: [6, 5, 4, 3]      # 按等级
phases:
  - phase: PRE_CAST          # 前摇
    duration: 0.3s
    can_cancel: true         # 可被打断
    effects: [play_animation, lock_facing]
  - phase: CAST             # 施法瞬间
    effects:
      - type: spawn_projectile
        projectile_id: 20001
        speed: 1200
        on_hit_effects:
          - type: damage
            formula: "0.6 * ATK + 0.4 * SP + 100"
          - type: apply_buff
            buff_id: 30001  # 燃烧 DOT
            duration: 3s
  - phase: POST_CAST        # 后摇
    duration: 0.2s
    can_cancel: false
```

**Effect 是组合式插件**（damage / heal / spawn / buff / dispel / teleport / push / pull / shield / summon / vfx），新技能 = 配表里组合 effect。

**Effect 执行器（DS 端）：**
- 每个 effect 类型实现一个 `IEffect` 接口，`Apply(context)` 纯函数
- context 包含 caster / target / position / level / skill_instance，确定性输入

**好处：**
- 策划自助开技能，不需要程序员写代码
- 行为可序列化、可回放、可在编辑器里预览
- 技能修复 / 数值调整可以**热更配表不发版**

**主流引擎参考**：LOL 的 BBLua、Dota2 的 KeyValues、王者的自研。

### Q5: 技能"瞬发但有飞行物"（如火球术）和"瞬间命中"（如点击式伤害）逻辑上怎么区分？

**A:**

**关键概念：技能效果发生的时间点 ≠ 玩家按键时间点。**

| 类型 | 时间线 | 实现 |
|------|--------|------|
| 瞬发（lock-on, hitscan） | 按键 → 命中即时算 | 在 phase=CAST 直接 Apply damage |
| 弹道（projectile） | 按键 → 飞行 → 命中算 | spawn 一个 ProjectileEntity，每帧推进位置，碰撞时 Apply |
| 延迟落点（meteor） | 按键 → N 秒后落 | 注册延迟事件 (frame + delay) → 到帧 Apply |
| 持续区域（黑洞）| 按键 → 区域存在 N 秒，每秒结算 | 一个 AreaEntity，每 tick 找区域内单位 Apply |
| 引导（持续施法） | 按键 → 每 tick 算一次，被打断停止 | caster 进入 channeling state，可被控打断 |

**所有"未来才发生"的效果必须可被取消 / 修正**：
- 释放者死亡 → 召唤物 / 弹道 / 引导都需要按规则处理（有的继续，有的消失）
- 目标进入无敌 / 闪现 → 弹道追踪策略：丢失目标 / 锁定原位 / 转其他目标
- DS 重启不应该影响这些（实际上 DS 不会重启，一局一进程）

## 二、确定性深挖

### Q6: 定点数怎么选位宽？Q16.16 够用吗？

**A:**

**Q16.16（int32）**：
- 整数部分 ±32768，小数精度 1/65536 ≈ 0.0000153
- MOBA 地图坐标范围通常 ±15000 → 整数部分够
- 速度 < 2000 unit/s → 够
- **问题**：乘法溢出！`a * b` 中间结果可能超 int32，必须 `(int64)a * b >> 16`
- **三角函数**：sin/cos 查表（4096 项足够）

**Q32.32（int64）**：
- 范围超大，乘法中间用 int128 才不溢出
- 内存 2 倍，cache 命中率下降
- 适合需要大坐标（开放世界）或高精度（物理）

**MOBA 通常选 Q16.16**，性能好，地图够用。少数大世界（如刀塔自走棋的某些模式）用 Q32.32。

**实现要点：**
```cpp
struct Fixed {
  int32 raw;
  static Fixed FromInt(int v) { return {v << 16}; }
  static Fixed FromFloat(float v) { return {(int32)(v * 65536.0f)}; }  // 仅用于配表加载，运行时禁用
  Fixed operator*(Fixed o) const {
    return { (int32)(((int64)raw * o.raw) >> 16) };
  }
  // ...
};
```

**严禁运行时把 Fixed 转回 float 再计算**——会引入平台差异。**整个逻辑层禁用 float**，渲染层（客户端）想用浮点随便。

### Q7: 你提到 sin/cos 查表，那角度精度怎么办？AOE 边缘判定会不会出问题？

**A:**

**典型方案：4096 项 sin 表，索引 = (angle * 4096 / (2π)) & 4095**，精度约 0.087°。

**AOE 边缘判定**：实际项目里 **AOE 半径用 Fixed，目标点位置也用 Fixed**，距离比较：

```cpp
bool inRange(Fixed cx, Fixed cy, Fixed tx, Fixed ty, Fixed r) {
  Fixed dx = tx - cx, dy = ty - cy;
  // 用平方比较，避免 sqrt
  return dx*dx + dy*dy <= r*r;
}
```

`dx*dx + dy*dy` 需要 Q32.32 中间值防溢出。

**扇形 AOE**（前向锥形）：
```cpp
// 判断 target 是否在 facing 方向 ± half_angle 内
Fixed dot = dx * facing_x + dy * facing_y;
Fixed cos_half = sin_table[half_angle_index + QUARTER];  // 查表
return dot >= cos_half * distance;
```

避免实际算 `acos`——把比较转成 `dot ≥ cos_threshold * length` 形式。

### Q8: 重新讲下"客户端预测 + 服务器回滚"在帧同步里怎么用？

**A:**

帧同步本身就是"客户端按收到的帧序列执行"，**没有传统意义的回滚**。但是有两种"局部预测"：

**1. 渲染插值（无回滚）**：
- 逻辑层 30Hz，渲染层 60+Hz
- 用 `lerp(prev_pos, cur_pos, t)` 平滑显示
- 玩家点了技能但帧还没广播 → 渲染层先播放释放动画的"前摇"
- 帧到了 → 渲染对齐到 DS 的判定（如果预测错就硬切，但前摇 0.3s 足够掩盖）

**2. 命令前瞻（Input Prediction）**：
- 玩家按了移动 → 客户端立刻渲染移动起步（不等下一帧广播）
- DS 收到指令 → 在 frame N 广播 → 全员包括自己执行到 frame N 时和本地预测对齐
- 如果延迟 100ms → 前瞻 3 帧 → 抵消感受到的延迟

**和 FPS 的状态同步回滚（lag compensation）的区别：**
- FPS：服务端在判定时把目标"回滚"到射击者按键时的位置，避免"我明明瞄准了"
- MOBA 帧同步：服务端不做回滚，**所有玩家共享同一帧的状态**，自然公平。代价是高延迟玩家感觉操作有 100ms 延迟

**优秀实现**：王者的 "100ms 抗延迟" 就是结合**乐观帧 + 前瞻 + 渲染插值**，让 80ms 内的延迟玩家完全无感。

## 三、AI / Bot

### Q9: 小兵 / 野怪 AI 怎么做？

**A:** **行为树（Behavior Tree）+ 黑板**，几乎是行业标配。

**典型小兵 AI 行为树：**

```
Selector (优先级)
├── [Sequence] 找最近敌方英雄（5 格内） → 攻击
├── [Sequence] 找最近敌方小兵（攻击范围内） → 攻击
├── [Sequence] 找最近敌方塔（攻击范围内） → 攻击
├── [Sequence] 检查是否被技能打中 → 切换仇恨到攻击者
└── [Default] 沿预设路径前进
```

**关键：行为树执行必须确定性**：
- 节点遍历顺序固定
- 选目标时如果多个目标同等优先级（距离一样），用 entity_id 做 tie-breaker
- 黑板上的"上次攻击者"用 frame 时间戳，不用 wall clock

**AI 节流：**
- 不是每帧 tick 所有 AI，按 entity_id 模 N 分到不同帧
- 远离玩家视野的野怪可以 1Hz tick
- 大批小兵共享决策（5 个一波小兵的 AI 一起算）

**性能：** 一局 60 个小兵 + 50 个野怪 + 0 个 bot，AI 总消耗 < 5% 帧时间。

### Q10: 玩家断线 AI 托管要做什么？和真人有什么差异？

**A:**

**目标**：托管 AI 不影响队友体验（不送、不挂机），但**绝对不能强于真人**（避免恶意挂机）。

**实现：**

```
玩家断线 10 秒 → DS 标记该单位为 ai_controlled
进入"保守模式"行为树：
  - 跟随队友（不孤立）
  - 攻击距离内的敌人（不主动追）
  - 血量 < 40% 撤退到塔下
  - 不主动用 R 大招（避免乱开团）
  - 偶尔（每 N 秒）补刀小兵保经济
```

**关键限制：**
- AI 决策只能用**和真人玩家相同的信息**（视野内 + 公开信息），不能开图
- AI 行为延迟随机化（150–300ms），不能 0ms 反应
- 玩家回连立刻交还控制权

**统计**：托管 AI 通常打不过中等以上水平真人，平均 KDA 接近 0/5/2，不会"演员级表现"。

---

### Q11: PVE 模式（人机局）的 Bot 和断线托管 AI 有什么区别？

**A:**

**断线托管**：低强度，保证队友体验。
**人机 Bot**：模拟真人，有难度分档（青铜/白银/...）。

**人机 Bot 设计要点：**

| 维度 | 实现 |
|------|------|
| 难度分级 | 反应延迟（300→100ms）、技能命中率（40%→90%）、补刀率（30%→80%）、决策周期（2s→0.5s） |
| 走位 | 预设巡逻路线 + 受击逃跑 + 团战时尝试侧翼包抄 |
| 出装 | 配表预设出装顺序，按等级解锁，根据敌方阵容选反制装 |
| 团战 | 检测附近敌我数量，3v3 以上判定团战，全员集中火力优先级目标 |
| 多样化 | 同一个英雄多套 Bot 行为模板，避免玩家秒识破 |

**绝对不要**：让 Bot 调用上帝视角 API、用真实属性数值偷偷加成、跨视野预判技能——这些行为玩家秒辨认出来，体验直接崩。

**实现框架**：Utility AI（多个 score 函数加权选最高分动作）比 BT 更适合人机 Bot，BT 易出现"机械感"。

## 四、网络深挖

### Q12: KCP 调参实战，nodelay / interval / resend / nc 分别什么意思？MOBA 怎么调？

**A:**

```c
ikcp_nodelay(kcp, nodelay, interval, resend, nc);
```

| 参数 | 含义 | MOBA 推荐 |
|------|------|----------|
| nodelay | 是否启用 nodelay 模式（影响 RTO 算法） | 1（启用） |
| interval | 内部 update 周期（ms） | 10（即 100Hz 重传检查） |
| resend | 快速重传：N 个 ACK 跳过即触发重传 | 2 |
| nc | no congestion control，关闭流控 | 1（MOBA 流量小，关掉省事） |

**额外：**
- `ikcp_wndsize(kcp, 256, 256)`：发送/接收窗口 256 包，MOBA 单玩家足够
- `ikcp_setmtu(kcp, 1400)`：避免 IP 分片（MTU 1500 留余量给 IP/UDP 头）
- 发送时尽量合并多个小包成一个 segment（KCP 自动做但要尽量减少 send 次数）

**MOBA 总流量**：单玩家 ~15KB/s 上行、~20KB/s 下行，KCP 完全过剩。**抗丢包**才是关键：5–10% 丢包下 KCP nodelay=1 + FEC 冗余比 TCP 快 5–10 倍。

**避坑：**
- 不要在 KCP 之上再套一层应用 ACK，浪费
- update 频率太低（50ms+）→ 重传等很久，玩家感受卡顿
- 多个玩家共用一个 socket 时，按 conv 路由

### Q13: 移动端弱网（地铁、电梯、4G→WiFi 切换），DS 怎么扛？

**A:** **多层抗丢包 + 长重连窗口 + 客户端缓冲。**

**1. 网络层：**
- KCP FEC 冗余包（10% 冗余抗 5% 丢包）
- 自适应码率：检测 RTT > 200ms 时减少非必要数据（如装备购买动画事件）

**2. 协议层：**
- 帧合并：DS 不一定每个 frame 都单独发包，可以 2–3 帧合一个 UDP 包（带宽友好，但增加延迟）
- 重要事件单独可靠通道（结算、断线、被踢）

**3. 会话层：**
- 重连窗口 60–180 秒（前面讲过）
- 客户端切网（IP 变化）→ session token 续接，不用重新连接 DS
- "网络异常" UI 显示 5 秒后才弹断线（避免短抖动也弹窗）

**4. 客户端缓冲：**
- 渲染层有 100–200ms jitter buffer，平滑抖动
- 输入预测 + 渲染插值（前面讲过）

**5. 体验降级：**
- 4G 模式：自动降画质，减少特效粒子（DS 不发某些观赏性事件）
- 切到流量警告：客户端缓存最近几帧的输入，到 WiFi 后批量发

**典型指标：王者在 5% 丢包 + 200ms RTT 下仍能正常对局，10% 丢包是体验拐点。**

### Q14: DS 主动踢人的几种情况？怎么和反作弊配合？

**A:**

| 触发 | 检测 | 处理 |
|------|------|------|
| 心跳超时 | 60s 没收到任何包 | 标 disconnect，进入重连窗口 |
| 指令异常 | 同一帧上传多次操作、指令字段越界 | 立刻断开 + 记 ban 日志 |
| 帧校验失败 | 客户端 frame_checksum 和 DS 不一致 | 几次后断开 + 标可疑 |
| 速度异常 | 客户端报告位置和服务端推算偏差大 | 强制同步 + 累计积分超阈值断开 |
| 关键事件不一致 | 客户端"我打死了 X"但服务端判定没死 | 拒绝客户端结果，记日志 |
| 反作弊 SDK 上报 | 客户端检测到外挂特征 | 标 ban，比赛结束才告知（避免恶意拼接） |

**踢人时机的考量：**
- **比赛中不立刻封号**：不公开告知，记日志，比赛后异步处理。避免外挂作者实时摸出反作弊规则。
- **极端情况**（恶意攻击 DS 协议）：立刻 ban session，但保留对局让其他人继续。

## 五、跨平台一致性

### Q15: PC + 移动 + 主机三端同局怎么保证一致？

**A:**

**首要原则：逻辑层完全确定性，三端跑同一份字节码。**

| 挑战 | 对策 |
|------|------|
| 不同平台浮点行为 | 全定点数，禁 float |
| 不同编译器 | 关键逻辑用 C++ 标准库 + 定点数，跨编译器一致 |
| 不同 CPU 字节序 | 全 little-endian（ARM/x86 默认），网络包用网络字节序解析 |
| 不同性能水平 | 服务器是权威，客户端只负责显示，性能差仅影响帧率不影响逻辑 |
| 不同输入方式 | 触屏/键鼠/手柄统一成 `LogicCommand`（移动、技能、停止），DS 不知道原始输入 |

**最难的是新增平台：**
- 任何确定性回归（如新 SDK 引入 SIMD 指令）必须用**录像回归测试**（跑历史录像，比对结果哈希）
- 新平台首发前**所有历史录像都要能在新平台跑出相同结果**

---

### Q16: 同一台 DS 不同帧大小的客户端（120fps 旗舰机 vs 30fps 千元机）怎么公平？

**A:**

**关键：渲染帧率和逻辑帧率分离。**

- DS 逻辑 30Hz，所有客户端必须按 30Hz 接收和执行
- 客户端渲染 30/60/120Hz 任选，仅影响动画平滑度
- 操作输入按"逻辑帧"采样：120fps 客户端每 4 个渲染帧产生 1 个逻辑帧输入

**手感差异：**
- 高帧率渲染感觉更"丝滑"（动画插值密），不等于操作更准
- 触屏 vs 键鼠的精度差异**远大于**帧率差异，所以**默认匹配只在同输入类型间**（触屏匹触屏，键鼠匹键鼠）

**例外**：竞技模式可能开放跨平台，需要平衡英雄数值（如键鼠版降低指向性技能命中率）。

## 六、压测 / 上线

### Q17: 怎么压测 DS？模拟 10 个真人玩家的开销很大吧？

**A:** **不用真客户端，用 headless 机器人。**

**机器人架构：**

```
压测进程 = N 个 goroutine / coroutine
每个 goroutine = 一个假玩家：
  - 建立 KCP 连接到 DS
  - 收 DS 帧广播 → 不渲染，只更新本地状态
  - 按行为模板生成指令（移动 / 释放技能 / 攻击）
  - 上报指令到 DS
```

**单机能模拟多少？** 一个普通 8 核机器轻松模拟 1000–5000 个假玩家（不渲染，开销极小）。**压一整局 = 1 个 DS + 10 个 bot，单机能跑 100 局并发。**

**压测指标：**
- DS 帧时间分布（P50/P95/P99/Max）
- DS CPU / 内存 / GC
- 网络抖动模拟（注入 50–200ms 延迟 + 5% 丢包）
- DS Allocator 拉起延迟
- 结算回调成功率

**关键场景：**
- 团战 10v10 全员释放大招同时段
- 极端：满图小兵 + 召唤物（持续推线场景）
- 网络抖动：丢包 → 大量重传 → KCP 发送队列堆积

我们项目有类似压测套路：[stress-test-2026-05-ephemeral-port.md](stress-test-2026-05-ephemeral-port.md)。

### Q18: DS 上线前必须跑的检查清单是什么？

**A:**

**P0（不通过禁止发版）：**

1. **录像回归**：100 局历史录像跑出相同哈希
2. **确定性 fuzz 测试**：随机生成 1000 局指令流，三台不同机器跑结果一致
3. **崩溃测试**：注入异常指令、超大包、畸形包，DS 不崩溃只断开恶意连接
4. **内存泄漏**：连续跑 100 局 DS 不退出（虽然实际一局一进程，但要测内存增长），无明显增长
5. **结算幂等**：DS 主动崩溃 + 重试结算，结算服只发一次奖励
6. **压测达标**：单 DS 跑标准对局 P99 帧时间 < 25ms（30Hz 阈值 33ms 留 25% 余量）
7. **跨平台一致**：iOS / Android / PC 三端跑同一对局，逐帧 checksum 一致
8. **断线重连**：随机断网 10–60s，所有玩家能完整重连追帧

**P1（强烈建议）：**

9. **降级演练**：拔 Redis、拔 Kafka，DS 行为可预期
10. **观战 SDK**：所有事件能被 OB 客户端正确回放
11. **新英雄回归**：所有新增英雄的所有技能至少打 10 局，无崩溃和异常数值

**上线策略：**
- 先内部测试服 → 抢先服（1% 玩家）→ 灰度大区 → 全量
- **必须有 5 分钟回滚预案**：DS 拉起脚本能瞬间切回旧版本镜像

### Q19: 线上 DS 进程崩溃了怎么办？玩家怎么挽回？

**A:**

**分级处理：**

| 影响 | 行动 |
|------|------|
| 单局 DS 崩溃 | DS 进程标 Failed → 通知 10 个玩家"对局异常" → 不扣段位 + 全员发补偿（金币/经验/钥匙） |
| 同一时段大量 DS 崩溃 | 触发熔断：DS Allocator 暂停发新局，告警值班 |
| 同版本 DS 100% 崩溃 | 立刻回滚到上一版本，下线问题镜像，事后查 coredump |

**关键基础设施：**
- DS 进程必须开 coredump（K8s Pod 用 hostPath 挂载 dump 目录）
- 每个 DS 进程独立 `matchId` 日志，便于复盘
- 崩溃前最后 N 帧的指令流缓存到磁盘（崩溃后重放定位）
- 结算服收到 DS 崩溃通知 → 触发"对局取消"流程：补偿钱包、移除战绩、退还消耗品

**避免怎么也救不了的崩溃：**
- 严禁 DS 直接 `assert(false)` 崩溃，应该 graceful fail：标对局 invalid + 发结算崩溃事件
- DS 之外的任何依赖（Redis/Kafka/Allocator）短暂不可用不能让 DS 崩，应该重试或降级

## 七、特殊场景

### Q20: 观战（OB）模式怎么做？延迟为什么必须比玩家慢？

**A:**

**模式：**

| 模式 | 延迟 | 实现 |
|------|------|------|
| 玩家自由观战（朋友视角） | 实时（同玩家） | 普通客户端连接 DS，但 input 被拒 |
| 第三方上帝视角 OB | **延迟 3 分钟** | OB 客户端从 DS 拉延迟流（或拉录像 + 实时追） |
| 大型赛事导播 OB | 延迟 30 秒 + 5 分钟 | 专用 OB 通道 |

**为什么 OB 必须延迟？**
- **防代打 / 防作弊**：实时 OB 会让对方教练通过 OB 给选手报点位
- **导播切镜需要时间**：抓团战切镜头要预判
- **回看高光时刻**：3 分钟延迟方便剪辑回放精彩镜头

**实现：**
- DS 把全状态流推到 OB 缓冲服务（延迟队列）
- OB 客户端从缓冲服务拉流，缓冲服务负责限流和延迟
- 全图视野 OB 流量大（10x 普通玩家），不能让大量 OB 直连 DS

### Q21: 重赛 / 暂停对局怎么实现？

**A:**

**暂停（赛事专用）：**

```
裁判 / 队长触发暂停 → DS 进入 paused 状态
- 所有 entity 停止 tick
- 计时器（buff 到期、技能 CD）冻结
- 客户端显示"暂停中"
- 玩家不能操作（仅可聊天）
恢复 → DS 退出 paused 状态，所有计时器从暂停时点继续
```

**关键：** 暂停时点的所有 entity 状态作为快照，恢复时按快照继续。**严禁** "暂停期间客户端记录的输入"被回放，必须全清。

**重赛（rematch）**：
- DS 不实际"重启"，而是结算当前局为 invalid → DS 退出 → 调度新 DS → 同 10 人进新局
- 段位、装备、金币全部按"未发生"处理（依赖结算服的"无效对局"流程）

**重连不算重赛**：玩家断线只是个体重连，整体对局继续。

### Q22: 5 人组队 vs 单排 vs 双排的体验平衡，DS 这层能做什么？

**A:** DS 一视同仁，**平衡责任在匹配服**（不让 5 黑 vs 5 单），DS 不区分。

但 DS 可以提供数据辅助：
- 每局结算时上报"队伍组队结构"（5 黑 / 3+2 / 2+1+1+1 / 全单），结算服分析胜率
- 5 黑 vs 5 黑的对局可以加少量 MMR 加成（鼓励组队）
- 单排玩家进 5 黑对面的局，结算时段位变化打折（保护单排心情）

**DS 自身**：完全无差别处理，玩家就是玩家，不知道谁和谁一队（除了同 team）。

### Q23: 给一个新人接手 MOBA DS，第一周让 TA 做什么任务最能上手？

**A:** 按"由浅到深"：

1. **Day 1**：本地拉一个 DS 进程，连上 10 个 bot 跑完一局，看完整结算流程
2. **Day 2**：在配表里加一个英雄的新技能（纯组合现有 effect），DS 端跑通
3. **Day 3**：实现一个新的 effect 类型（如"5 秒内每秒回 3% 血"），加单测确保确定性
4. **Day 4**：写一个录像回归测试，跑 10 个历史录像确认通过
5. **Day 5**：定位一个真实的 P99 慢帧（看 trace 找瓶颈），尝试优化

**核心目标：理解"确定性、ECS、配表驱动、帧调度、录像"这 5 个支柱**。其他都是这 5 个的衍生。

**避免**让新人一上来就改核心调度循环或网络层 —— 那是高危区，破坏一处确定性全员崩。
