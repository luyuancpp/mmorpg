# 跨场景玩家消息投递

## 问题
Scene A 想给玩家发消息，但玩家可能不在 Scene A 上。需要一套统一的消息投递机制。

## 两种情况

### 情况 1：玩家不在本 Scene（跨场景投递）

**旧方案**：Centre 转发 → 复杂，Centre 成为瓶颈/单点
**新方案**：通过 Kafka 投递到玩家所在 Scene 的 topic

**关键边界问题 — 路由过程中玩家切换场景**：
- 消息已放入 Kafka（目标 Scene B 的 topic）
- 路由过程中玩家从 Scene B 切换到 Scene C
- Scene B 收到消息时玩家已不在 → 消息丢失或投递到错误场景

**解决方向 — 消息重要性分级（via protobuf option）**：
- 在 RPC service 定义中用 `option` 标记消息重要性级别
- **重要消息**（如交易结果、奖励发放、状态变更）：
  - 需要确认投递 / 失败重试
  - Scene B 发现玩家已离开 → 重新查询 PlayerLocator → 转投到 Scene C
  - 或回退到持久化队列（DB/Redis），玩家下次上线/进入场景时补发
- **非重要消息**（如聊天气泡、特效通知、非关键 UI 提示）：
  - 丢了就丢了，不需要重试
  - Scene B 发现玩家不在，直接丢弃

### 情况 2：玩家在本 Scene（本地投递）

**待决策**：直接同步处理？还是也统一放入玩家消息队列？

**直接处理（同步路径）**：
- 优点：延迟最低，简单直接
- 缺点：与跨场景路径的处理逻辑不一致，可能导致消息顺序不一致

**统一消息队列（异步路径）**：
- 优点：本地和跨场景走同一条路径，逻辑统一，消息顺序有保证
- 缺点：本地消息多了一次队列中转的开销
- 符合"位置透明"原则

**倾向**：统一走玩家消息队列更稳健，与跨服架构的"位置透明"原则一致。
但对于极低延迟需求的消息（如移动同步）可以特殊处理走直接路径。

## 设计要点

1. **Protobuf option 标记重要性**：在 service/rpc 方法定义上加 option，代码生成时可据此生成不同的投递策略
2. **投递失败处理策略因重要性而异**：重要消息保证至少一次投递（at-least-once），非重要消息 best-effort
3. **玩家消息队列**：每个玩家在所在 Scene 上维护一个有序消息队列，统一处理入口
4. **与现有 Kafka 架构整合**：复用 `gate-{gate_id}` topic 模式，可增加 `scene-{scene_id}` topic 或 `player-msg-{partition}` topic
