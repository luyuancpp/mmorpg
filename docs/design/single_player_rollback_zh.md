# 单人回档 — MMO/MOBA 微服务架构

> **讨论日期**：2025-03-25
> **背景**：MMO/MOBA 微服务架构回档设计讨论

## 问题

微服务架构下，玩家数据分散在多个服务（背包、任务、邮件、货币、公会、好友、排行榜等）。将单个玩家回档到之前的状态，需要跨所有服务进行协调恢复。

## 推荐方案：应用层快照 + 按需回档

### 每个服务暴露 Export/Import RPC
- `ExportPlayerData`：导出该服务中玩家的完整状态（protobuf 字节）
- `ImportPlayerData`：用提供的数据覆盖玩家状态（幂等）

### 快照服务
- 定期（以及关键事件触发时：登录、大额交易、维护前）调用各服务的 ExportPlayerData
- 存储复合快照：`player_id | snapshot_time | service_name | data_blob`
- 保留策略：最近 N 天的登录快照 + 每日定期快照
- 存储：专用表/数据库，与线上数据隔离

### 回档流程
1. 将玩家踢下线（必须——在线时不可回档）
2. 查找最近且 ≤ 目标时间戳的快照
3. 按依赖顺序对每个服务调用 ImportPlayerData
4. 写入审计日志（操作人、原因、目标时间、受影响服务）
5. 向 GM/客服工具返回结果

### 关键设计要点
- **必须离线**：回档前踢下线，防止写冲突
- **幂等性**：ImportPlayerData 必须支持安全重试
- **不逆转外部影响**：发给他人的邮件、公会捐献、交易——只回滚玩家自身状态
- **粒度**：支持全量回档和按服务回档（例如只回档背包，不动任务）
- **审计追踪**：每次回档记录谁/何时/为何/范围
- **快照隔离**：与线上数据库分离，防止故障级联
- **服务顺序**：按依赖顺序恢复（通常无循环依赖）

### 兜底方案：数据库级 PITR
- 使用 MySQL binlog 时间点恢复到影子库
- 从影子库中提取目标玩家在目标时间戳的行数据
- 导回线上数据库
- 运维成本较高；最适合作为灾难恢复的最后手段

### 最佳实践：混合策略
- 常规回档 → 应用层快照（快速、精确、可自动化）
- 灾难恢复 → MySQL PITR（覆盖广、运维成本高）

## 防复制：非对称回档问题

### 问题描述
玩家 A 将物品交易给玩家 B。如果只回档 A，则 A 和 B 都拥有该物品 → 物品复制。
适用于所有双向操作：交易、邮件附件、公会仓库、拍卖行、队伍分配。

### 解决方案：回档 ≠ 时光倒流，回档 = 异常损失补偿

**核心原则：有正常转移记录的物品/货币不恢复。**

### 必需基础设施
1. **物品 UUID**：每个物品实例拥有全局唯一 ID（SnowFlake），支持精确追踪。
2. **交易流水日志**：所有双向操作写入 `transaction_log`，通过 Kafka → DB：
   - tx_id、时间、类型（交易/邮件/拍卖/公会仓库）、来源玩家、目标玩家、item_uuid、数量、货币变化
3. **全局物品索引**：item_uuid → 当前持有者查询（仅在回档物品时查询）

### 回档流程（修订版）
1. 将玩家踢下线
2. 加载目标时间的快照
3. 对比快照与当前状态
4. 对每个"丢失"的物品/货币：检查 transaction_log
   - 有正常转移记录 → 不恢复（接收方仍持有）
   - 无转移记录（Bug/异常）→ 恢复
5. 生成"可恢复物品"报告供 GM 审核
6. GM 批准 → 执行选择性恢复
7. 写入审计日志

### 备选策略（取舍）
- **级联回档**（连带回档所有交易方）：风险太大，影响面不可控
- **软回档**（仅邮件补偿，不覆盖数据库）：最安全，用于常规客服工单
- **物品存在性检查**（UUID 仍存在于世界中则跳过恢复）：比完整流水分析更简单但精度较低

### 核心原则
> 单人回档不是"时光倒流"——而是"异常损失补偿"。
> 任何有合法交易流水记录的转移，一律排除在恢复范围之外。

## Bug 利用 / 物品复制 — 精确回收

### 场景
玩家发现 Bug 并利用它生成物品/货币，然后通过交易/邮件/拍卖扩散。

### 处理层次
1. **冻结**：立即封号止损
2. **追踪污染链**：从 transaction_log 追踪每个脏 item_uuid 和货币流向（多级扩散：A→B→C→...）
3. **精确回收**：
   - 物品还在某人身上 → 按 UUID 移除
   - 已卖给 NPC → 扣除等价货币
   - 在拍卖行上 → 下架并删除
   - 脏货币已花费购买其他物品 → 扣除那些物品或等价物
4. **无辜玩家处理**：
   - 知情同谋 → 回收 + 可能封号
   - 通过正常交易购买的不知情买家 → 移除物品但退还其支付的金币
   - 扩散面太广无法追踪 → 金额小则接受损失；严重则紧急维护 + 全服回档
5. **核弹选项**：经济体系污染过重 → 全服回档（MySQL PITR）

### 四种场景汇总
| 场景 | 操作 | 基础设施 |
|------|------|----------|
| 正常物品丢失（意外） | 软回档（邮件补偿） | 交易流水日志 |
| 结构性数据损坏 | 快照恢复（真正回档） | 定期快照 + Export/Import RPC |
| Bug 利用复制 | 精确回收（反向追踪） | 交易流水日志 + 物品 UUID + 污染链 |
| 全服经济崩溃 | 全服回档 | MySQL PITR / 全服快照 |

**以上四种 + 补缴系统（见下文）均依赖：物品 UUID（SnowFlake） + 交易流水日志。**

## 补缴系统（小额利用的柔性回收）

### 适用场景
- 小额 Bug 利用，未大规模扩散，玩家非恶意
- 纯货币/代币复制（非唯一性物品）
- 希望在不封号、不影响玩家体验的前提下回收

### 工作原理
1. GM 判定玩家通过 Bug 获得了 X 数量的货币
2. 创建 `player_debt` 记录：玩家欠 X 数量的 currency_type
3. 每次玩家获得该货币时（任务奖励、交易收入、系统发放...）：
   → 系统在统一的 `AddCurrency` 入口拦截
   → 扣除 min(incoming_amount, remaining_debt) 到补缴池
   → 玩家收到剩余部分
4. 持续直到欠款全部还清

### 数据模型
```
player_debt:
  player_id | currency_type | owed_amount | paid_amount | reason | gm_operator | created_at | completed_at
```

### 实现要点
- **必须 hook 到唯一的统一货币增加函数** — 如果货币可以通过绕过 hook 的多条代码路径添加，则扣款容易被规避
- 伪代码：
```
AddCurrency(player, type, amount):
  debt = GetPlayerDebt(player, type)
  if debt > 0:
    deduct = min(amount, debt)
    actual_gain = amount - deduct
    ReduceDebt(player, type, deduct)
  else:
    actual_gain = amount
  player.currency[type] += actual_gain
```

### 适合与不适合
- **适合**：同质化货币（金币、钻石、代币），小额，非恶意玩家
- **不适合**：唯一性物品（必须使用基于 UUID 的精确回收）、大额（欠款过大无法自然还清）、反复/恶意利用者（应封号）

### 相比硬性扣款的优势
| | 硬性扣款（直接扣除） | 补缴系统 |
|-|----------------------|----------|
| 玩家体验 | 突然损失，容易引发投诉 | 渐进式，低摩擦 |
| 负数余额风险 | 已花费可能扣成负数 | 永远不会为负，只从收入中扣除 |
| 需要封号吗？ | 通常不需要，但持续利用可能需要 | 不需要，修复 Bug 后挂欠款即可 |
| 实现复杂度 | 简单 | 中等 — 需要统一的货币入口 |

## 货币系统架构决策

### 结论：货币放在 C++ 玩家实体上，Go 只负责持久化和 GM 操作

**原因**：战斗中的货币操作（买药、技能代价、修理费等）要求零延迟，RPC 到 Go 服务 2–20ms 的往返不可接受。

### 架构
- 登录时：从 DB 加载到 `CurrencyComp`（C++ 节点内存中）
- 游戏中：纯内存读写
- 定期及下线时：通过 Kafka → go/db → MySQL 刷入脏数据
- 补缴 hook 放在 C++ 的 `AddCurrency` 统一入口内

### Go 服务职责范围（仅限离线/外围场景）
- 充值到账
- Web 商城购买
- 邮件附件货币
- GM 工具
- 以上最终都通知 C++ 节点执行实际货币变更
