# MOBA 服务器面试 Q&A — DS（Dedicated Server / 战斗服）

**Created:** 2026-05-18

> 范围：单局对战的战斗服（DS / Game Server / Battle Room），即玩家进入"载入界面 → BP/选英雄 → 进入峡谷 → 出结算"那一段。
> 非战斗外围（大厅 / 匹配 / 房间 / 战绩 / 段位 / 聊天 / 商城）见另一篇 [moba-non-ds-server-interview-qa.md](moba-non-ds-server-interview-qa.md)。

---

## 一、整体定位

### Q1: MOBA 的 DS 和 MMO 的 Scene 有什么本质区别？为什么不能复用一套架构？

**A:**

| 维度 | MOBA DS | MMO Scene |
|------|---------|-----------|
| 生命周期 | **短**：15–40 分钟一局，结束即销毁 | **长**：常驻，玩家进进出出 |
| 玩家规模 | 固定 10 人（5v5）+ ~50 单位 AI | 数百到数千在线 |
| 进入方式 | 匹配系统配对后**一次性**全员进入 | 玩家随时按需进入/退出 |
| 状态权威 | DS 即权威，结束后状态丢弃 | Scene 是权威，状态持久化到 DB |
| 帧模型 | **定 tick（30/30/60Hz）+ 帧同步或锁步广播** | 变频 tick + AOI 状态同步 |
| 重连 | 必须支持断线 → 回放追帧 | 重连即新登录 |
| 确定性 | **强**（录像、反作弊、防作弊回滚都依赖） | 弱（服务端权威即可） |
| 扩缩容 | **按局拉起**（K8s Job / Agones / 进程池） | 长驻进程，灰度替换 |

**核心结论**：MOBA DS 是 **"一局一进程 / 一局一容器"** 模型，不是常驻服。架构应围绕 *快速冷启动 + 帧调度 + 录像 + 结算落库* 设计，不能套用 MMO 的常驻 Scene。

---

### Q2: 整体拓扑画一下，DS 在什么位置？

**A:**

```
                            ┌──────────────────┐
                            │  Matchmaker      │  (匹配服)
                            └────────┬─────────┘
                                     │ MatchFound(10 players, mode, map)
                                     ▼
                            ┌──────────────────┐
                            │  Room Service    │  (开房 / BP / 选英雄)
                            └────────┬─────────┘
                                     │ AllReady → AllocateDS
                                     ▼
                            ┌──────────────────┐
                            │  DS Allocator    │  (Agones / 自研 K8s Operator)
                            │  - 从 Ready 池摘一个 DS
                            │  - 注入 matchId / token / 名单
                            └────────┬─────────┘
                                     │ DSReady(ip:port, sessionToken)
                                     ▼
   ┌──────────┐  UDP/KCP/QUIC   ┌────────────┐    gRPC      ┌──────────────┐
   │ Client × │ ──────────────► │  DS 进程   │ ───────────► │  结算服      │
   │   10     │ ◄────────────── │  (一局)    │              │  战绩/段位   │
   └──────────┘                 └─────┬──────┘              └──────────────┘
                                      │ 录像 / 关键事件
                                      ▼
                                ┌──────────────┐
                                │  Replay/OSS  │
                                └──────────────┘
```

DS **不挂任何 L4/L7 LB**，客户端直连。Allocator 把 `ip:port + sessionToken` 直接下发给客户端，套路同我们项目的 gate 直连原则（见 [gate-load-balancing-design.md](gate-load-balancing-design.md)）。

## 二、网络与协议

### Q3: MOBA DS 用 TCP 还是 UDP？为什么？

**A:** **UDP（KCP / ENet / Steam GameNetworking / QUIC datagram）为主**。

- MOBA 对**延迟**和**抖动**极敏感，操作指令丢一个 100ms 内必须重发，TCP 的队头阻塞 + RTO 几百毫秒不可接受。
- 关键控制信令（结算、断线、心跳）可以走 TCP/可靠 UDP 通道。
- 我们项目 MMO 用 TCP（muduo）的原因是 AOI 同步对短暂卡顿容忍度高；MOBA 不能照抄。

**典型分层：**

```
应用层：操作指令（unreliable / unreliable-ordered）
        状态快照 / 关键事件（reliable-ordered）
        结算 / 断线 / 心跳（reliable）
传输层：KCP / ENet / GameNetworking，自带 ARQ + 拥塞 + 冗余 FEC
```

**加分点：**
- 公网环境 KCP `nodelay=1, interval=10, resend=2, nc=1` 是常见低延迟档
- FEC 冗余包在丢包率 5% 以上明显优于纯 ARQ
- 移动网络抖动大，要做 jitter buffer

### Q4: 服务端给客户端推送状态，是帧同步还是状态同步？

**A:** 两条主流路线，按品类选：

| | 帧同步（Lockstep） | 状态同步 |
|---|---|---|
| 代表 | LOL、王者荣耀、Dota2 | Overwatch、Valorant、Apex |
| 服务端工作 | 只转发指令 + 推帧号 | 跑完整模拟 + 下发状态快照 |
| 带宽 | 极低（每帧仅 10 人的指令） | 较高（每帧广播单位状态） |
| 确定性 | **必须**（差一个浮点全员脱钩） | 不强求 |
| 防外挂 | 弱（客户端有全图状态） | 强（服务端只发可见的） |
| 反作弊战雾 | 难（要再做一层 FOW 过滤） | 天然支持 |
| 回放 | 录指令即可，极小 | 录快照，较大 |
| 断线追帧 | 重发指令序列 | 重发关键快照 + delta |

**MOBA 主流是帧同步**，因为：
1. 10 人小队，指令量很小（每人每秒 ~30 指令 × 几字节）
2. 客户端必须做精准动画/技能预判，状态同步延迟感强
3. 录像极小（一局 < 1MB）

**代价**：必须做战争迷雾的**视野裁剪在客户端**（虽然服务端有完整状态，但服务端不广播状态，客户端只用自己视野内的指令结果渲染）。**反外挂必须配合服务端权威校验**（关键事件如击杀、推塔、Roshan，DS 重算一遍）。

---

### Q5: 帧同步具体怎么实现？解释一下"乐观帧 vs 悲观帧"。

**A:**

**悲观帧（强同步）**：每帧 DS 等齐所有 10 个客户端的指令才广播下一帧。
- 优点：完全同步，无回滚
- 缺点：**最慢的玩家拖死全场**（一个 200ms 的玩家 → 全员 5Hz）

**乐观帧（弱同步 / 我们叫"帧广播"）**：DS 按固定 tick 推帧号，客户端指令带 `frameId`，到点没收到就当**空指令**。
- 优点：稳定 30Hz
- 缺点：迟到的指令要么丢弃，要么客户端做**预测 + 回滚**

**主流（王者 / LOL）走乐观帧**：
```
DS：每 33ms broadcast Frame(N, cmds=[a,b,c,...])
    丢失指令 → 该玩家此帧 = 空操作
客户端：按 frame 推进逻辑层
        渲染层用插值/外推平滑
```

**加分点：**
- 服务端要保证 broadcast 是**原子的**：同一帧所有 10 人收到的指令集合必须一致
- 一个 frame 的指令收齐后，DS 计算一个 `frame_checksum`（哈希）一起下发，客户端比对，不一致 → 上传日志 + 强制断线
- 重连：客户端报上次 ack 的 `frameId`，DS 从录像里 reflood 缺失帧

### Q6: 帧同步的"确定性"具体指什么？哪些坑会破坏确定性？

**A:**

**定义**：相同的初始状态 + 相同的指令序列，在不同机器 / 不同时刻执行，得到**逐字节一致**的结果。

**常见破坏源（必背）：**

1. **浮点跨平台不一致**：x86 SSE vs ARM NEON, `sin/cos` 不同库微差 → **改定点数（Q16.16 / Q32.32）**，所有数学走自己的 `FixedMath`
2. **`std::unordered_map` 遍历顺序**：哈希依赖地址，不同进程不同 → 用 `std::map` 或自定义有序容器
3. **未初始化内存 / `padding`**：序列化前清零
4. **多线程**：DS 内部必须**单线程跑游戏逻辑**，IO 线程只投递消息
5. **时间戳作为输入**：`std::chrono::now()` 不可入逻辑层，只能由 frame tick 推进
6. **随机数**：必须 seeded PRNG（如 xoshiro256），种子从 matchId 派生，全 10 端一致
7. **容器扩容时机不同**：保留 reserve，或干脆用对象池
8. **指针 / 地址作为 ID**：换成稳定 entity id
9. **`std::sort` 的稳定性**：相同 key 顺序可能不同 → 用 `stable_sort` 或加 tie-breaker
10. **三方库**：物理引擎（Box2D / PhysX）默认是非确定的，必须用确定性版本或自研

**加分点：自动化校验**
```
DS 每 N 帧（如 60）对全状态算 hash，客户端也算
客户端上传 hash 不一致时 → DS 主动 dump 双方状态做 diff
```

## 三、Tick / 帧调度

### Q7: DS 怎么调度 tick？30Hz 还是 60Hz？怎么保证不漂移？

**A:**

主流 MOBA 逻辑层 **30Hz**（33.3ms），渲染层 60+Hz 客户端插值。守望先锋等 FPS 是 60Hz 逻辑。

**调度套路：**

```cpp
auto next = steady_clock::now();
while (running) {
    next += 33ms;                      // 累加而不是 now()+33ms，避免漂移
    pollNetwork();                     // 收 10 人指令
    advanceFrame();                    // 推进逻辑帧
    broadcastFrame();                  // 下发
    auto sleep = next - steady_clock::now();
    if (sleep > 0ms) sleep_for(sleep);
    else logLag(sleep);                // 来不及了，记录卡顿
}
```

要点：
- `next += interval` 累加，**绝对时间基线**，避免 sleep 不准引起的 drift
- 卡顿（sleep < 0）要记日志 + 监控告警；**严禁** "spiral of death"（持续追帧把 CPU 100%）
- 必要时**跳帧追赶**：连续 3 帧延迟 > 1 帧就跳一帧并通知客户端
- DS 主线程 **CPU 亲和性绑核**，关闭超线程同核占用

---

### Q8: DS 一个进程跑一局还是多局？

**A:** **看资源密度和隔离需求。**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 一进程一局 | 完全隔离，崩溃只影响一局；K8s/Agones 友好；冷启快 | 进程数多，内存基线浪费 |
| 一进程多局（如 4 局） | 内存共享代码段，节省 200MB+/进程；调度灵活 | 一局卡死影响其他局；难做亲和性绑核 |

**主流商业 MOBA 选一进程一局**（王者 / LOL），原因：
1. 崩溃域小，单局 OOM/SIGSEGV 只丢一局
2. Agones / 自研 Operator 模型天然按 Pod 调度
3. 升级灰度按局粒度，新版本只影响新开的局
4. 录像、profiling、coredump 直接对应一局

**自研小作坊**或国内 H5/小游戏 MOBA 会选多局共进程，省机器。

---

### Q9: DS 一局结束后怎么回收？冷启动要多久？

**A:**

**两种池化策略：**

1. **Warm Pool**（推荐）：预启一批 DS 进程，停在 "Ready" 状态等待 Allocate
   - 优点：Allocate → 玩家进入 < 1 秒
   - 缺点：占机器，要预测高峰
2. **冷拉起**：匹配成功才 K8s 拉 Pod
   - 优点：完全弹性
   - 缺点：拉起 3–10 秒，玩家在 Loading 等

**回收：**

```
结算完成 → DS 上报 Result → 落库 → DS 退出（Exit 0）
                                    ↓
                              K8s 回收 Pod / Agones 标记 Allocated→Shutdown
                                    ↓
                              HPA / Operator 维持池水位
```

**严禁 "重置状态复用同一进程"**——内存碎片、引用泄漏、确定性 PRNG 状态污染都会出问题。**一局一进程，结算即死**。

**加分点：Agones**（Google 开源的 K8s 游戏 DS 编排器）就是为这个场景设计的，有 Ready/Allocated/Shutdown 状态机和 SDK。

## 四、视野 / 战雾

### Q10: MOBA 的战争迷雾（FOW）服务端怎么做？要不要服务端裁剪？

**A:** **必须服务端裁剪**，否则反外挂全失守（地图透视挂的根源）。

**实现：**

```
地图按 32×32 像素划格子 → 网格视野图
每个单位有 sight_radius（白天）/ sight_radius_night（夜晚）
视野来源：英雄、小兵、防御塔、守卫（眼）、特殊技能

每 tick：
  对每个 team（己方=2 队）：
    visible_grids[team] = ∪ (unit.position 半径内格子)  for unit in team.units
  对每个单位 u：
    visible[team][u] = (u.position in visible_grids[team]) 或 u.is_truesight_target
  广播给 team 的状态 = filter(all_units, visible[team])
```

**优化：**
- 视野格子用 bitmap（512×512 bit = 32KB / team），位运算合并极快
- 单位移动 < 1 格不重算视野
- 草丛 = 视野被裁的特殊格子（在草丛外的单位看不到草丛内）
- 真实视野（True Sight）覆盖隐身 / 隐形
- 视野变化用 delta：上一帧可见 vs 这一帧可见 → enter/leave 事件，不全量推

**帧同步模式的特例**：客户端有全图指令，所以服务端推帧时**不能省略指令**（省了对不上帧），但客户端在**渲染层做 FOW 裁剪**——加上服务端定期下发的关键事件（击杀、推塔）校验。
这是帧同步反外挂最大的痛点：**地图透视外挂在客户端层很难根治**，只能靠机器学习识别异常操作 + 客户端代码加固。

## 五、断线 / 重连 / 录像

### Q11: 玩家断线 30 秒重连，DS 怎么追帧到当前？

**A:**

DS 维护：
- `frames[]`：每帧的指令集合（帧同步）或状态快照（状态同步）
- `keyframes[]`：每 N 帧（例如 300 帧 = 10 秒）一个全状态快照

**重连流程：**

```
Client → DS: Reconnect(matchId, lastAckFrame=12345, sessionToken)
DS:
  1. 校验 token + 玩家在该局名单
  2. 找到 lastAckFrame 之后最近的 keyframe（如 frame 12300）
  3. 下发 KeyFrameSnapshot(12300) + Frames[12301..currentFrame]
  4. 客户端：load snapshot → fast-forward 重放剩余帧 → 跟上
  5. DS 切换该 sessionId 到新连接，继续正常推帧
```

**关键点：**
- 追帧期间客户端**渲染不显示**（黑屏 + Loading），逻辑全速回放
- 服务端持有的指令缓冲必须够长（覆盖最大允许重连窗口，如 3 分钟）
- DS 重发**幂等**：重发同一段帧客户端能正确处理（行号去重）

**断线但不退出（挂机）**：DS 视该玩家指令为空操作，继续推帧。AI 托管由客户端断线判定后通知 DS 启用。

### Q12: 录像（Replay）怎么存？多大？

**A:**

**帧同步录像 = 指令流 + 初始状态**：

```
ReplayFile {
  header: {matchId, version, mapId, mode, players[], startTime, seed}
  initialState: {team_compositions, items, ...}
  frames: [{frameId, cmds: [{playerId, cmdType, payload}, ...]}, ...]
  events: [{frameId, type, ...}]  // 关键事件（击杀/推塔/Roshan），用于无指令快速预览
  checksums: [{frameId, hash}]    // 用于回放校验
}
```

**体积：** LOL/王者一局录像 200KB–2MB，普通 zstd 压缩后更小。可直接传 OSS/S3，URL 给客户端按需下载。

**状态同步录像** 要存关键快照 + delta，体积大一个量级（10–50MB），通常只存"高光时刻"。

**用途：**
1. 玩家回看（客户端用本机引擎 + 录像重跑）
2. 反作弊审计（GM 服务器跑 headless DS 重放）
3. 训练数据（AI / 平衡性分析）
4. 比赛 OB / 全局回放

**加分点**：录像版本号必须和 DS 二进制绑死，跨版本不保证可回放（确定性会因数值/算法改动失效），所以必须按版本号归档。

## 六、反外挂 / 防作弊

### Q13: 帧同步天然把全图状态发到客户端了，怎么防地图透视外挂？

**A:**

帧同步 + 战争迷雾**是反作弊最难的组合**。多层防御：

1. **客户端加固**：代码混淆、内存加密、反调试、反注入（VMP / 自研壳）
2. **关键计算服务端权威**：击杀判定、推塔、经济、技能命中——客户端报告，DS 用自己算的结果覆盖客户端，不一致就记日志
3. **行为模型**：DS 每帧记录所有玩家操作，结算后送 ML 模型识别"前知后觉"（眼朝草丛预判走位、技能预释放）
4. **客户端举报回放复审**：被举报的对局 GM 重放 + 模型评分
5. **网络包指纹**：合法客户端的网络包时序模式，外挂修改后会有特征

**不要承诺"100% 防住"**——帧同步模式下完全消灭地图透视外挂在公开场景里没有商业产品做到了，只能压低普及率。

---

### Q14: 服务端权威校验具体怎么做？给一个技能命中的例子。

**A:**

```
帧 N: 客户端 A 释放技能 Q（指向技能，目标点 (100, 200)）
       上报 cmd = {type: CastSkill, skill: Q, target: (100, 200), predictedHit: [B, C]}
DS 收到，加入帧 N 的指令集合
帧 N+1: DS 在自己的世界推进
       根据 A 的位置 + Q 的弹道公式 → 实际命中 = [B]（C 因为护盾 / 闪现没命中）
       DS 的状态更新：B 扣血、C 不扣
       帧 N+1 的下发指令包含 "skill_result: {hits: [B], misses: [C]}"
客户端：根据 DS 下发的 hits/misses 修正本地预测（如果错了，C 的血回滚）
```

DS 永远是**判定权威**，客户端的 `predictedHit` 只是优化体验，不参与判定。这套就是"帧同步 + 服务端权威"组合，业界 K8s + 王者都用。

## 七、性能 / 可观测

### Q15: 一台物理机能跑多少局 DS？怎么算？

**A:**

**核心约束是 CPU 单核性能**（帧同步主线程不能并行），不是核数。

经验公式（5v5 MOBA）：
- 一局 DS 主线程占用：**0.05 – 0.15 CPU 核心**（即一个核心可以跑 6–20 局）
- 内存：80–200MB / 局
- 带宽：上行 15KB/s/玩家 × 10 = 150KB/s/局；下行 20KB/s/玩家 × 10 = 200KB/s/局

**一台 32 核 / 128GB / 10Gbps 服务器：**
- CPU 上限：32 × 10（局/核）= **320 局**
- 内存上限：128GB / 150MB = **~870 局**
- 带宽上限：10Gbps / 4Mbps（一局上下行合计）= **~2500 局**

**CPU 是瓶颈**，规划 250–300 局/物理机比较稳。要绑核（一局一物理核），关闭 turbo boost 引起的频率漂移（影响确定性的 sleep 精度）。

### Q16: 一局 DS 卡顿（某帧推进超过 33ms），怎么定位？

**A:**

**埋点：**

```
每帧记录：
  - frame_id
  - cmd_count（这帧处理了几个指令）
  - phase_duration: {network_poll, advance, broadcast}
  - active_units / active_skills（活跃单位/技能数）
帧耗时 > 30ms 触发慢帧日志
连续 3 帧慢 → 上传完整 trace
```

**常见原因（按频率）：**
1. **GC / 内存分配高峰**——用对象池消除，C++ 用 arena allocator
2. **AOI 重建** / 视野脏标记爆炸 —— 增量更新，不全量
3. **技能广播 N²**——技能范围内单位数大时遍历未做空间索引
4. **日志 I/O 同步阻塞**——异步落盘，主线程只投递
5. **网络收包暴涨**——拉黑攻击源，做单连接 rate limit
6. **物理引擎 / 寻路** —— 限制每帧 budget，超出部分摊到下一帧

**线上工具：**
- 进程自带 `/debug/pprof`（如 Go）或 perf 火焰图（C++）
- DS 关键事件打到 OpenTelemetry，分布式追踪到结算服

## 八、特殊话题

### Q17: 大乱斗 / 自走棋 / 8 人混战，DS 设计要改什么？

**A:**

| 模式 | 玩家数 | 关键差异 |
|------|--------|----------|
| 5v5 MOBA | 10 | 标准 |
| 大乱斗 | 10 | 单条路 → 视野简单，技能密度高，单局更短（10 分钟） |
| 自走棋 | 8 | **没有实时操作**，每回合 30 秒备战 + 10 秒战斗，DS 主要是回合状态机 + 玩家间棋盘 1v1 自动战斗 |
| 8 人混战 | 8 | 没有队伍概念，视野/伤害判定都是个人 |

自走棋 DS 特殊：**回合制 + 战斗预算**——备战阶段 DS 几乎空载（玩家操作棋盘下棋），战斗阶段两两配对跑确定性自动战斗，可以**并行**跑 4 场（因为是独立 1v1）。

### Q18: 跨服 / 跨大区怎么处理 DS？

**A:** 三种模型：

1. **DS 全球池**：所有大区的玩家进入同一个全球 DS 池，按延迟匹配机房。需要全球同步的匹配服。
2. **大区独立**：每个大区有自己的 DS 池，跨大区匹配走"中立机房"DS 池（如港澳台 vs 东南亚 → 香港机房）
3. **就近原则**：匹配成功后选 10 人**平均延迟最小**的机房 DS

主流是 2 + 3 组合。DS 本身无状态，**机房选择只影响延迟，不影响逻辑**。

### Q19: DS 怎么和外围（结算服、战绩、段位）交互？

**A:** DS 是"局内权威，局外只发结果"。

```
DS 在局结束时：
  POST /match/result {
    matchId, duration, winnerTeam, players: [{id, kda, gold, hero, items, ...}],
    keyEvents: [...],   // 击杀链、推塔时间
    replayUrl,
  }
结算服：
  - 写入战绩库（MySQL / TiDB / 自研列存）
  - 计算段位变化（ELO / Glicko-2）
  - 发奖（金币 / 经验 / 战令）→ 走异步消息（Kafka）给各业务服
  - 通知好友 / 公会 / 战绩订阅
```

**幂等性**：结算用 `matchId` 做幂等键，重复提交（DS 重试）不能重复发奖。Kafka 用 `matchId` 做 key，分区有序消费。

DS 退出策略：**确认结算服 200 OK 后才 Exit 0**，否则进入指数退避重试，超过 5 分钟未成功 → dump JSON 到本地 emergency queue + 报警 + Exit 1。
