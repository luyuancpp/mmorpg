# 率土之滨 SLG 游戏服务器主程序面试 Q&A

**Created:** 2026-04-01

## 一、大地图与寻路

### Q1: 率土之滨的大地图规模大约 1200×1200，服务端寻路你会选什么算法？为什么不用 NavMesh？

**A**: 格子地图用 **A\*** 足够，配合 **JPS（Jump Point Search）** 在方格图上可获得 10-30x 加速。不用 NavMesh 是因为 SLG 地图本质是离散格子，不是连续地形，NavMesh 的三角剖分在格子图上没有意义，反而增加复杂度。

对于跨大半地图的远距离寻路，可以用 **HPA\*（Hierarchical Pathfinding）**：将地图分成 chunk（如 32×32），先做粗粒度跨 chunk 路由，再做 chunk 内精细寻路，避免在百万格子上跑单次 A\*。

**加分点**：
- 寻路必须在服务端做（防作弊），客户端只预测展示
- 地图障碍物变化（建城/拆除）只需更新受影响 chunk 的局部连通图
- 相同起终点 + 相同地图版本号 → 路径缓存命中直接返回

---

### Q2: 行军过程中地图上有建筑被拆了，路径变了怎么办？

**A**: 两种策略：

1. **被动重新寻路**：部队到达被阻断的格子时发现不可通行，触发重新寻路。实现简单，但可能出现部队到了才发现走不通的情况。

2. **主动失效通知**（推荐）：地图障碍物变化时，检查该格子上是否有行军路径经过（格子上维护 `passing_armies` 列表）。如果有，给这些部队的拥有者发"路径失效"事件，服务端自动重新寻路并更新剩余路径。

实际项目中用方案 2，因为 SLG 地图变化频率低（建城/拆除不频繁），影响的行军数量有限，重新寻路的开销可控。

## 二、行军同步

### Q3: 地图上可能同时有数万支部队在行军，服务端怎么处理？要不要每帧 tick 推进每支部队的位置？

**A**: **不需要每帧 tick**。用基于时间的惰性求值：

```
current_tile = path[min(floor((now - t_start) * speed), len(path) - 1)]
```

行军创建时：
1. 计算完整路径
2. 算出到达时间，注册一个**定时器事件**（到达时触发战斗/驻扎/采集）
3. 沿路径注册视野进入/离开事件的时间点

中途任何时刻查询部队位置，直接从公式算，O(1)。5 万支部队在行军 ≠ 5 万个 tick 实体，服务端只有到达事件和视野事件需要处理。

**客户端同步**：只需收到 `{path, start_time, speed}`，本地插值动画。加速/召回时服务端推一次更新事件即可。

### Q4: 两支敌军在行军途中相遇怎么检测？不 tick 的话怎么知道它们碰上了？

**A**: **时空碰撞检测**。每个格子维护时间占用表：

```
Tile(x,y).occupants = [{army_id, enter_time, leave_time}, ...]
```

新行军注册时，遍历路径上每个格子，检查时间段是否和已有敌军重叠：

```
重叠条件: new_enter < existing_leave && existing_enter < new_leave
```

如果碰撞，在最早重叠时刻调度一个战斗事件。**注册时一次性检测完毕**，不需要逐帧轮询。

当某支部队被召回或速度变化时，需要清除旧的占用记录并重新注册。

**具体示例**：

```
时间轴:     t=0    t=1    t=2    t=3    t=4
部队A路径:  [A0] → [A1] → [A2] → [A3] → [A4]
部队B路径:              [B0] → [B1] → [B2] → [B3]

假设 A2 和 B1 是同一个格子:
  部队A在 t=2 到达, t=3 离开 → [2, 3)
  部队B在 t=2 到达, t=3 离开 → [2, 3)
  2 < 3 && 2 < 3 → 重叠！在 t=2 触发战斗
```

实现代码思路：

```cpp
struct TileOccupancy {
    struct Entry {
        uint64_t army_id;
        double   enter_time;
        double   leave_time;
    };
    std::vector<Entry> occupants;
};

// 新行军注册
for (int i = 0; i < path.size(); i++) {
    double enter_t = start_time + i / speed;
    double leave_t = start_time + (i + 1) / speed;
    
    Tile& tile = map[path[i]];
    for (auto& existing : tile.occupants) {
        if (IsEnemy(existing.army_id, new_army_id) &&
            enter_t < existing.leave_time && 
            existing.enter_time < leave_t) {
            double battle_time = max(enter_t, existing.enter_time);
            ScheduleBattle(battle_time, existing.army_id, new_army_id, path[i]);
            return;
        }
    }
    tile.occupants.push_back({new_army_id, enter_t, leave_t});
}
```

## 三、战斗系统

### Q5: 率土之滨的战斗是实时的还是回合制的？服务端怎么处理？

**A**: **回合制自动战斗，确定性模拟**。

战斗是纯函数：`BattleReport = SimulateBattle(attacker, defender, seed, terrain)`

- 相同输入永远产生相同结果
- 服务端运行完整模拟，生成**战报**（事件序列）发给客户端回放
- 客户端不需要有战斗逻辑，只需要动画播放器

**确定性约束**：
- 禁止浮点数，全用整数/定点数（`damage = atk * 120 / 100`）
- 统一 seeded PRNG（如 xoshiro256），调用顺序固定
- 禁止依赖系统时间、内存地址、STL 无序容器遍历顺序

---

### Q6: 战报怎么设计？客户端怎么回放？

**A**: 战报是**事件日志**，不是录像：

```protobuf
message BattleReport {
    uint64 battle_id       = 1;
    uint64 random_seed     = 2;
    ArmySnapshot attacker  = 3;
    ArmySnapshot defender  = 4;
    repeated Round rounds  = 5;
    BattleResult result    = 6;
}

message Round {
    uint32 round_number          = 1;
    repeated BattleAction actions = 2;
}

message BattleAction {
    ActionType type        = 1;   // NORMAL_ATTACK, SKILL, BUFF, DEBUFF, HEAL, DEATH
    uint32 source_slot     = 2;   // 发起者位置 (0-5)
    uint32 target_slot     = 3;   // 目标位置
    uint32 skill_id        = 4;   // 触发的战法ID
    int32  damage          = 5;   // 伤害/治疗量
    int32  source_hp_after = 6;   // 行动后血量（自包含）
    int32  target_hp_after = 7;
    repeated BuffEvent buffs = 8;
    bool   is_critical     = 9;
    bool   is_dodged       = 10;
}
```

关键设计：每个 Action **自包含 hp_after**，客户端不需要自己算伤害，展示永远和服务端一致。

客户端按顺序逐条播放动画，支持 1x/2x/跳过。因为 Action 含状态快照，任意位置可 seek。

**为什么不用输入重模拟？**

| | 事件日志（SLG 首选） | 输入重模拟 |
|---|---|---|
| 战报内容 | 完整 Action 序列 | 只存 BattleInput + seed |
| 客户端要求 | 只需动画播放器 | 必须内嵌完整战斗逻辑 |
| 数据量 | ~1-5 KB/场 | ~200 bytes/场 |
| 跨版本兼容 | 旧战报永远可播放 | 逻辑变更后旧 seed 结果不同 |
| 安全性 | 客户端无战斗逻辑可逆向 | 客户端有完整逻辑 |

选择事件日志：长期存储（联盟战史）、跨版本兼容、降低逆向风险。

### Q7: 多路部队集结攻城，战斗怎么处理？

**A**:
1. 集结令发起后，各参与部队向目标行军
2. 所有部队到达后（或等待超时），合并为攻方阵容
3. **按到达顺序排成攻击队列**，依次或分批与守方交战
4. 守方连续作战，血量不回满，体现"车轮战"消耗

服务端用一个 **SiegeSession** 管理整场攻城的多轮战斗状态。

## 四、视野 / 迷雾

### Q8: 率土之滨有战争迷雾，玩家只能看到自己视野范围的信息。服务端怎么实现？

**A**: **格子级 observer 计数器**：

```
每个 Tile 维护: observer_count[player_id] → int

建城/部队进入视野范围格子: count++
离开: count--
player 能看到该格子 ⟺ count > 0
```

行军时用**滑动窗口增量更新**：部队从格子 A 移到格子 B，只需要：
- 新前沿格子（B 的视野边缘新增部分）count++
- 旧尾部格子（A 的视野边缘丢失部分）count--

视野半径固定（如 3 格），用预计算的**偏移表**批量处理，不需要每次算圆形区域。

---

### Q9: 这跟 MMORPG 的 AOI 有什么区别？能不能复用？

**A**: 不能复用，问题模型完全不同：

| | MMORPG AOI | SLG AOI |
|---|---|---|
| 核心问题 | "谁在我旁边"（以实体为中心） | "我能不能看到那里"（以格子为中心） |
| 坐标系 | 连续浮点 | 离散整数格子 |
| 数据结构 | 九宫格哈希 / 十字链表 | 每格子 observer 计数器 |
| 更新频率 | 高频（每秒十几次移动） | 低频（行军每秒移动 1 格） |
| 推送触发 | 实体状态变化 → 推给附近玩家 | 格子状态变化 → 推给 count>0 的玩家 |

**MMORPG 九宫格示例**：
```
┌─────┬─────┬─────┐
│     │     │     │   地图切成大区块（如 100x100）
├─────┼─────┼─────┤   玩家 P 在中间格
│     │ [P] │     │   P 能看到周围 9 格内所有实体
├─────┼─────┼─────┤
│     │     │     │
└─────┴─────┴─────┘
```

**SLG observer 计数示例**（视野半径 = 2）：
```
      3   4   5   6   7
  3 [ 1 ][ 1 ][ 1 ][ 1 ][ 1 ]
  4 [ 1 ][ 1 ][ 1 ][ 1 ][ 1 ]    observer_count[player_A] = 1
  5 [ 1 ][ 1 ][城][ 1 ][ 1 ]    表示"player_A 能看到这个格子"
  6 [ 1 ][ 1 ][ 1 ][ 1 ][ 1 ]
  7 [ 1 ][ 1 ][ 1 ][ 1 ][ 1 ]
```

### Q10: 玩家拖动地图浏览大地图，服务端怎么处理？会不会产生大量请求？

**A**: 分两层：

1. **视野内数据**（己方城池/部队视野范围内）：服务端主动推送变化事件，客户端缓存
2. **视野外但视口内**（玩家拖地图看到但没有视野的区域）：显示迷雾/只显示地形和公开建筑

客户端拖动地图时，按**视口（viewport）** 向服务端请求格子数据。服务端做**权限过滤**：
- observer_count > 0 的格子：返回完整信息（部队、资源等）
- observer_count = 0 的格子：只返回地形 + 公开信息（城池名称、等级）

**防刷优化**：
- 客户端按 Region（如 32×32）为单位请求，不逐格子请求
- 服务端对视口请求做频率限制
- 地形数据客户端本地缓存，只请求动态数据

## 五、性能与架构

### Q11: 一张 1200×1200 的大地图，服务端怎么扛住？要不要分服？

**A**: 单服内用 **Region 分片**：

- 将地图切成 N×N 个 Region（如 32×32 格为一个 Region）
- 每个 Region 由独立线程/协程处理，减少全局锁
- 跨 Region 行军时做 handoff（转交给目标 Region 处理）

内存估算：
- 1.5M 格子 × 128 bytes/格 ≈ 200 MB，完全放得进内存
- 5 万支行军部队的路径和定时器 ≈ 几十 MB
- 单服总内存 < 1 GB

**分服策略**：率土是赛季制，每个赛季是独立的"服"。单服 5000-10000 人，一台机器可以跑多个服。不需要像 MMORPG 那样做跨服无缝大世界。

---

### Q12: 大量定时事件（行军到达、建筑完成、资源采集完成）用什么定时器？

**A**: **Hashed Timing Wheel（哈希时间轮）**。

为什么不用优先队列（堆）：
- 堆插入/删除是 O(log N)，N 可能到十万级
- 时间轮插入/删除是 O(1)

时间轮配置：
- 精度：100ms 一个 slot（SLG 不需要毫秒级精度）
- 一层轮：3600 slots = 6 分钟一圈
- 超过 6 分钟的事件放溢出链表，轮转时降级插入

实际场景：
- 行军到达：几十秒到几分钟
- 建筑升级：分钟到小时（这种超长的直接存数据库，不放在内存定时器里）

### Q13: 如果让你设计整体服务架构，一个"服"里有哪些进程/服务？

**A**:

```
客户端
  │
  ▼
Gate (网关层) ── 负责连接管理、协议解析、消息路由
  │
  ▼
┌─────────────────────────────────────────┐
│            逻辑层 (可水平扩展)              │
│                                         │
│  MapService     ── 大地图、行军、视野、格子  │
│  BattleService  ── 战斗模拟 worker pool     │
│  BuildService   ── 建筑建造/升级定时器       │
│  ArmyService    ── 部队编制、武将、征兵       │
│  AllianceService── 联盟、外交、集结          │
│  ChatService    ── 聊天频道                │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│              数据层                       │
│  Redis  ── 在线状态、排行榜、热数据缓存       │
│  MySQL  ── 持久化（玩家、战报、联盟）         │
│  Kafka  ── 跨服务异步消息、战斗结算通知       │
└─────────────────────────────────────────┘
```

关键设计选择：
- MapService 是有状态的（持有整张地图内存），**不做水平扩展**，做 Region 级并发
- BattleService 是无状态的纯计算，可以水平扩展 worker
- 玩家状态由 Gate 路由到 MapService（玩家绑在地图上，不像 MMORPG 会切场景）

## 六、数据持久化与一致性

### Q14: 玩家下线了，他的城池和部队还在地图上，数据怎么管理？

**A**: SLG 的核心特点是**玩家离线后世界仍在运转**。

- 所有地图实体（城池、部队、资源田）常驻内存
- 玩家上线只是"打开了观察窗口"，不是"加载实体"
- 定时写盘（Write Behind）：脏数据标记 + 定时批量落库
- 关键操作（战斗结算、资源变化）即时写 WAL/binlog

**下线玩家的城被攻打**：
- 正常触发战斗流程
- 战报写入玩家战报列表
- 上线时推送未读战报 + 状态变化摘要

---

### Q15: 服务器崩溃重启，几万支行军中的部队怎么恢复？

**A**: 行军数据持久化核心字段：`{army_id, path, start_time, speed, state}`

重启后恢复流程：
1. 从 DB 加载所有 `state=MARCHING` 的行军记录
2. 用 `now - start_time` 算出当前应该在路径的哪个位置
3. 如果已经过了到达时间 → 直接执行到达逻辑（补算战斗等）
4. 如果还在途中 → 重新注册剩余路径的定时器和视野事件
5. 重建每个格子的 occupancy 和 observer_count

**关键**：因为行军位置是从公式算的（不是 tick 积累的），所以丢失中间状态不影响正确性。这是惰性求值模型的最大好处之一。

## 七、安全与防作弊

### Q16: SLG 游戏主要有哪些作弊手段？怎么防？

| 作弊手段 | 防御措施 |
|---------|---------|
| 透视（看到迷雾后面的信息） | 服务端做视野权限过滤，未观察的格子不下发详细数据 |
| 加速行军 | 行军速度服务端计算和校验，客户端无法修改 |
| 伪造战斗结果 | 战斗在服务端运行，客户端只收战报 |
| 刷资源 | 所有资源变更服务端计算，校验产出速率 |
| 协议重放 | 请求加时间戳 + 序列号，服务端校验 |
| 多开/脚本 | 操作频率检测 + 行为模式分析 |

核心原则：**服务端是唯一权威**。客户端发的是"意图"（我要从 A 出兵到 B），不是"结果"（我的部队现在在 C）。

## 八、高级话题

### Q17: 赛季结束后几千万条战报和玩家数据怎么处理？

**A**:
- 赛季结束后数据**冻结**（只读）
- 热数据（排行榜、联盟战史摘要）迁移到归档库
- 完整战报走冷存储（S3/OSS），按需拉取
- 玩家核心数据（武将、道具）迁移到新赛季的初始状态
- 旧赛季的 MapService 可以关停释放资源

### Q18: 如果要做跨服战（多个赛季服的联盟打架），架构上怎么扩展？

**A**:
- 开一个独立的"战场服"，有自己的地图和 MapService
- 参战玩家的部队数据**快照复制**到战场服（不是引用原服数据）
- 战场服独立运行战斗逻辑
- 战斗结果通过消息队列同步回各原服（奖励、伤兵恢复等）
- 战场服是临时的，活动结束后销毁

**不要做**：让两个服的 MapService 直接通信——耦合太重、故障域不隔离。
