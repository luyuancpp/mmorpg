# Proto-Gen Naming Migration Checklist

This document defines the current naming boundary after moving from the historical name `pbgen` to the user-facing name `proto-gen` and the canonical internal name `protogen`.

## Current Policy

- Use `proto-gen` in user-facing docs, tasks, launch configs, and terminal help.
- Use `protogen` for canonical on-disk source and module names:
  - `tools/proto_generator/protogen/`
  - `tools/proto/protogen/`
  - `tools/docs/protogen/`
- Primary user-facing binary is `proto-gen.exe`; retain `pbgen.exe` as a compatibility copy.
- Keep `pbgen` as a compatibility alias in commands, binary copies, and legacy environment variables.
- Treat generated headers/comments like `Code generated by protogen` as generator-owned output.

## Migration Goals

- Reduce confusion for new contributors.
- Preserve existing local scripts and CI flows during transition.
- Avoid partial renames that split runtime behavior from documentation.

## Safe, Low-Risk Changes

- Update README files and AGENTS guidance to prefer `proto-gen`.
- Rename VS Code task labels and launch labels to `proto-gen`.
- Add `proto-gen-build` and `proto-gen-run` aliases while keeping `pbgen-build` and `pbgen-run`.
- Keep help output and examples consistent with the user-facing name.

## Medium-Risk Changes

- Add wrapper scripts that expose only `proto-gen` to new users.
- Update archived log/readme wording where the path remains `docs/protogen/`.

Before shipping medium-risk changes, verify:

- VS Code tasks still work from a clean checkout.
- `dev_tools.ps1 -Command proto-gen-build` and `proto-gen-run` work without manual env setup.
- Existing users invoking `pbgen-build` / `pbgen-run` are not broken.
- No CI or local automation assumes only `pbgen.exe` exists.
- `PROTOGEN_ENABLE_PPROF=1` works, and old `PBGEN_ENABLE_PPROF=1` still behaves the same.

## High-Risk Changes

- Remove the compatibility copy `pbgen.exe`.
- Remove the compatibility aliases `pbgen-build` and `pbgen-run`.
- Stop accepting `PBGEN_ENABLE_PPROF`.
- Change Go module/package/import paths from `protogen/...` to a different internal naming scheme.
- Change generated output banners from `Code generated by protogen` without validating generator diffs.

Before any high-risk rename, verify:

- `go build ./...` succeeds in `tools/proto_generator/protogen/`.
- All internal imports have been updated consistently.
- Launch configs, scripts, and tasks are updated together.
- Any generated-file diffs are reviewed as intentional, not incidental churn.
- Historical references needed for local toolchains are replaced or explicitly preserved.

## Recommended Rollout Order

1. Keep current state: user-facing `proto-gen`, internal `protogen`, compatibility `pbgen`.
2. Validate scripts, tasks, and local docs against the `protogen` directory layout.
3. Keep compatibility wrappers or duplicate outputs while existing automation still expects `pbgen.exe`.
4. Only then consider removing legacy aliases in a dedicated validation pass.

## Stop Conditions

Stop the migration and revert to the last stable compatibility layer if any of the following occurs:

- CI or local generation scripts fail because they assume a hard-coded `pbgen` path.
- Go imports diverge across files and produce partial-rename breakage.
- Generated outputs change in unexpected ways outside naming-only scope.
- Team workflows still depend on historical paths that were not audited.

## Remaining Intentional `pbgen` References

The following `pbgen` references are currently expected and should not be treated as cleanup misses:

- Compatibility paths and directories:
  - `tools/proto_generator/protogen/`
  - `tools/proto/protogen/`
  - `tools/docs/protogen/`
- Binary and environment names:
  - `proto-gen.exe`
  - `pbgen.exe`
  - `PROTOGEN_ENABLE_PPROF`
  - `PBGEN_ENABLE_PPROF`
- Compatibility commands and help text:
  - `pbgen-build`
  - `pbgen-run`
- Internal source-level identifiers:
  - Go module/import paths like `protogen/...`
  - package-local comments or logger paths under `tools/proto_generator/protogen/`
- Generated or archived content:
  - checked-in generated outputs under `generated/`
  - historical run logs and extracted fatal lines under `tools/docs/protogen/`

These categories should only be changed as part of a dedicated compatibility migration, not during routine documentation cleanup.

## Recommendation

For now, the repository is in the correct state:

- Human-facing name: `proto-gen`
- Internal canonical name: `protogen`
- Compatibility name: `pbgen`

Do not rename disk paths or package imports until there is a separate migration task with explicit validation time allocated.
